# Spec: 云舱 AirCabin V1 Core

## Background
云舱 AirCabin 面向经常乘坐飞机出行的用户，重点解决飞行途中“飞行信息记录 + 航班社交 + 相册整理”三类需求。产品需优先支持 Android 端，并在弱网、离线和飞行模式下仍保证核心功能可用。

## Users
- 高频差旅用户：关注飞行效率、统计数据、航司与航线复盘
- 旅行记录型用户：重视飞行足迹、年度飞行报告与旅途照片整理
- 机上碎片时间利用型用户：希望在长途飞行中高效清理相册、查看航线、打发时间
- 同航班轻社交用户：希望在有边界的前提下与同舱乘客交流拼车、目的地建议、转机经验

## In Scope
- 登机牌/航班信息导入，支持手动录入、拍照识别、截图识别、后续预留邮件导入
- 当前航班卡片、2D/3D 航线模拟、飞行阶段展示、剩余时间与 ETA
- 本次飞行统计、月度/季度/年度飞行次数、飞行时长、里程与航司分布
- 本地相册读取、按时间/截图/相似图/大视频等维度筛选与批量删除
- 同航班同舱位聊天室，支持文字消息、表情、话题标签、举报、拉黑、敏感词过滤
- 个人飞行档案、年度飞行报告卡片、权限与隐私设置
- 离线缓存、弱网降级、关键数据本地优先存储

## Out of Scope
- 航空级实时导航或对外宣称真实空中位置定位
- 首版开放式陌生人广场、私聊或跨航班社交网络
- 照片云备份、云端图像识别、照片内容推荐
- iOS 同步交付
- 航司会员积分、值机、改签等航旅交易能力

## Functional Requirements

### FR1. 登机信息导入
系统必须支持用户通过拍照、截图或手动录入方式创建航班记录。

Acceptance criteria:
- 用户可识别或录入航班号、日期、起飞地、目的地、舱位、航空公司、座位号
- OCR 解析后，系统必须展示字段确认页
- 低置信度字段需要高亮提示用户修改
- 导入成功后自动生成本次航班卡片并缓存本地

### FR2. 当前航班与航线模拟
系统必须在首页和航班页展示当前航班状态，并支持基于计划数据的航线模拟。

Acceptance criteria:
- 用户可查看起点、终点、预计航线、模拟位置、当前阶段、剩余时间、ETA
- 系统支持 2D 航线图，V1 预留 3D 视图开关
- 无实时数据时必须明确显示“模拟中”
- 已缓存航班在飞行模式下仍可展示基础航线和统计数据

### FR3. 飞行时间与年度统计
系统必须沉淀当前航班和历史航班的统计数据。

Acceptance criteria:
- 用户可查看当前航班已飞时长、剩余时长、总时长
- 用户可查看月度、季度、年度飞行次数、总时长、总里程
- 用户可按国内/国际、航司、航线维度查看分布
- 统计页支持图表展示与年度报告卡片生成

### FR4. 本地相册整理与批量删除
系统必须支持用户读取本地媒体资源并按高价值维度筛选、预览和删除。

Acceptance criteria:
- 用户可按时间、截图、大视频、相似照片、连拍等维度筛选
- 用户可快速多选、滑动选择、全选当前分组
- 删除前必须支持预览和二次确认
- 删除成功后展示释放空间结果
- 图片和视频分析优先在端上执行，不默认上传云端

### FR5. 飞行档案与年度报告
系统必须将导入的航班数据沉淀为个人飞行档案。

Acceptance criteria:
- 用户可查看历史航班列表与详情
- 用户可生成年度飞行报告卡片
- 用户可查看最常乘坐航司、最常飞行航线、飞行城市分布
- 删除某条航班记录后，相关统计应重算

### FR6. 同航班同舱位聊天室
系统必须在网络可用时为同航班同舱位用户提供轻量匿名聊天室。

Acceptance criteria:
- 仅航班与舱位验证通过的用户可加入对应房间
- 聊天室至少支持文字、表情、话题标签
- 系统必须支持举报、拉黑、敏感词过滤、频控和房间生命周期管理
- 飞行途中网络不可用时，聊天室需降级为不可进入或仅显示草稿提示

### FR7. 弱网与离线策略
系统必须保障核心功能在弱网或离线场景下可用。

Acceptance criteria:
- 已缓存的航班卡片、历史统计、相册整理能力在无网场景可打开
- 所有关键数据实体先本地存储，再异步同步
- 写操作需支持失败重试和离线队列
- 无网时必须展示可理解的降级说明

### FR8. 隐私、安全与合规
系统必须对照片、登机信息、位置信息、聊天内容等敏感数据提供最小必要保护。

Acceptance criteria:
- 登机牌敏感字段默认脱敏展示
- 相册图片不默认上传云端
- 聊天身份对房间内仅展示匿名名称
- 用户可查看权限用途并主动清除本地数据
- 敏感操作必须有确认反馈和日志记录能力

## Non-Functional Requirements
- 首页和航班页在已缓存数据情况下首屏目标 <= 2 秒
- 相册索引完成后，分类切换响应目标 <= 500ms
- 客户端需覆盖加载态、空态、错误态、离线态
- 关键接口需预留鉴权、幂等与风控字段
- 聊天服务需支持房间级频控与消息审核扩展

## Primary Screens
- 首次导入 / OCR 结果确认页
- 首页
- 航班页
- 飞行统计页
- 相册整理页
- 聊天室页
- 飞行档案页
- 个人中心 / 权限设置页

## User Stories
- 作为高频差旅用户，我希望在飞行模式下也能查看当前航班进度，这样我能随时知道大概飞到哪和多久到达。
- 作为旅行用户，我希望系统自动汇总全年飞行数据，这样我可以生成属于自己的年度飞行报告。
- 作为机上用户，我希望快速筛出截图、大视频和相似照片并批量删除，这样我能利用飞行时间整理手机空间。
- 作为同航班乘客，我希望在有网络时进入同舱聊天室，这样我可以低门槛交流拼车或目的地建议。

## Open Questions
- 首版是否接入第三方真实航班状态供应商，还是仅使用计划航班模拟
- 聊天室首发是否白名单灰度，以控制审核成本和网络不确定性
- 年度飞行报告是否首版即支持分享落地页和图片导出
